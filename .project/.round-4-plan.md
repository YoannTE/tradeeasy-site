# Plan Round 4 — Dashboard admin & Gestion TradingView

## Approche

Round 4 est principalement du travail Payload CMS admin. Les collections (Video, Testimonial,
TradeScreenshot, PromoCode) existent deja — le CRUD natif Payload fonctionne via /admin.
Le vrai travail est :

1. Dashboard admin custom avec metriques (MRR, abonnes)
2. Vue "Acces en attente" avec bouton d'action
3. Notification email admin a chaque inscription

Strategie : 1 seul agent suffit (tout est backend Payload).

## Agent "admin" — Dashboard, TradingView access, notifications

### Tache 1 : Dashboard admin custom

Payload permet d'ajouter des composants custom au dashboard via `admin.components.afterDashboard`
ou via une vue custom dans payload.config.ts.

Creer un composant admin custom `components/admin/dashboard-stats.tsx` :

- Affiche 4 metriques : Total abonnes actifs, MRR (Monthly Recurring Revenue), Abonnes en essai, Acces en attente
- Utilise l'API Payload pour recuperer les donnees (fetch /api/subscriptions + /api/users)
- MRR = (nb abonnes mensuels _ 49) + (nb abonnes annuels _ 349/12)
- Style simple, cartes avec chiffres grands

Enregistrer le composant dans payload.config.ts admin.components.

Alternative plus simple si le composant custom Payload est complexe :
Creer une page admin dediee `app/(payload)/admin/dashboard-stats/page.tsx` accessible via /admin/dashboard-stats

### Tache 2 : Vue "Acces en attente" + bouton "Acces accorde"

Option A (recommandee) : Utiliser les vues de liste Payload natives avec filtres pre-configures.
La collection Users a deja le champ tradingviewAccessStatus. Le admin peut filtrer manuellement.

Option B (meilleure UX) : Creer un custom admin view ou un afterDashboard component.

Implementation recommandee — Hook afterChange sur Users :

- `collections/hooks/on-tradingview-access-granted.ts`
- Quand tradingviewAccessStatus passe de "pending" a "granted" :
  1. Set tradingviewAccessGrantedAt = new Date()
  2. Trouver la Subscription trial du user et set trialActivatedAt = new Date()
  3. Envoyer email "Acces accorde" au user (via Resend ou console.log en dev)
- Ce hook se declenche quand l'admin change manuellement le statut dans le panel Payload

L'admin workflow :

1. Va dans /admin/collections/users
2. Filtre par tradingviewAccessStatus = "pending"
3. Ouvre le user
4. Change le statut de "pending" a "granted"
5. Sauvegarde → le hook se declenche → email envoye + trialActivatedAt set

### Tache 5 : Notification admin a l'inscription

Hook afterChange sur Users :

- `collections/hooks/on-new-user-created.ts`
- Quand un nouveau user est cree (et que c'est pas l'admin seed) :
  1. Envoyer un email a admin@admin.com (ou ADMIN_EMAIL env var)
  2. Contenu : "New subscriber: {email} — TradingView: {tradingviewUsername} — Waiting for access"
  3. En dev : console.log au lieu d'email

### Utilitaire email

Creer `lib/email/send-email.ts` :

- Fonction sendEmail({ to, subject, html })
- Si RESEND_API_KEY est placeholder → console.log("[EMAIL]", to, subject) et return
- Sinon → utiliser Resend pour envoyer

Creer `lib/email/templates/` :

- `access-granted.ts` : template HTML pour "Your TradingView access is now active!"
- `new-subscriber-admin.ts` : template HTML pour notifier l'admin

### Organisation des collections dans le panel admin

Mettre a jour payload.config.ts pour grouper les collections :

```ts

  // ...existing config
  meta: { titleSuffix: '— SimplifyPro Admin' },
}
```

Et ajouter `admin: { group: 'Subscribers' }` ou `admin: { group: 'Content' }` dans chaque collection.
