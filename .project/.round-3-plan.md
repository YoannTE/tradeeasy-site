## Approche

Round 3 est le coeur du business. Les taches sont fortement couplees (auth + Stripe + subscriptions).
On ne peut PAS les paralleliser completement car l'inscription depend de Stripe, et les webhooks
dependent de la logique abonnement.

Strategie : 2 agents avec des responsabilites claires.

- Agent "auth" : tout ce qui est authentification frontend + formulaires
- Agent "stripe" : tout ce qui est Stripe (Setup Intent, webhooks, subscriptions, promo)

IMPORTANT : Stripe et Resend ne sont PAS connectes en dev (pas de vraies cles API).
Le code doit etre fonctionnel mais testable sans services externes :

- Stripe : utiliser des variables .env (STRIPE_SECRET_KEY, STRIPE_PUBLISHABLE_KEY, STRIPE_WEBHOOK_SECRET)
  avec des valeurs placeholder. Le code verifie que les cles existent avant d'appeler Stripe.
- Resend : utiliser RESEND_API_KEY avec valeur placeholder. En dev, logger l'email au lieu de l'envoyer.

## Agent "stripe" — Stripe, Webhooks, Subscriptions, Promo

### Tache 4 : Integration Stripe Setup Intent

`lib/stripe/stripe.ts` :

- Initialiser le client Stripe avec STRIPE_SECRET_KEY
- Export de l'instance Stripe

`lib/stripe/create-checkout.ts` :

- Fonction createTrialSubscription(email, tradingviewUsername, priceId?) :
  - Cree un Stripe Customer si pas deja fait
  - Cree une Stripe Subscription avec trial_period_days: 7
  - Retourne le client_secret pour le Setup Intent
  - Sauvegarde le stripeCustomerId dans le User Payload

`app/api/stripe/create-subscription/route.ts` :

- POST endpoint appele par le formulaire d'inscription
- Verifie que l'utilisateur est authentifie
- Appelle createTrialSubscription
- Retourne le client_secret

### Tache 5 : Webhooks Stripe

`app/api/webhooks/stripe/route.ts` :

- Lire le raw body avec request.text() (AVANT tout parsing)
- Verifier la signature avec stripe.webhooks.constructEvent()
- Handler pour chaque event :
  - `checkout.session.completed` : confirmer l'inscription
  - `invoice.payment_succeeded` : mettre a jour Subscription status → active
  - `invoice.payment_failed` : mettre a jour Subscription status → payment_failed
  - `customer.subscription.updated` : syncer le statut (cancel, plan change)
  - `customer.subscription.deleted` : mettre a jour status → cancelled
- Idempotence : verifier stripeEventId avant de traiter (skip si deja vu)
- Chaque handler met a jour la collection Subscription via Payload
- Logging des events recus

`lib/stripe/webhook-handlers.ts` :

- Fonctions separees pour chaque type d'event
- Chaque handler : find Subscription by stripeSubscriptionId, update fields, update lastSyncedAt

### Tache 6 : Logique abonnement

`lib/stripe/subscription-logic.ts` :

- Fonctions pour la logique metier :
  - createTrialSubscription(userId, stripeSubscriptionId) : cree Subscription avec status trial
  - activateSubscription(subscriptionId) : status → active
  - handlePaymentFailed(subscriptionId) : status → payment_failed, note grace period start
  - cancelSubscription(subscriptionId) : status → cancelled
  - isSubscriptionActive(userId) : retourne boolean (trial ou active)

Integration avec le signup :

- Quand un user s'inscrit : creer le User Payload, creer le Stripe Customer, creer la Subscription (status: trial)
- Hook Subscription beforeChange : empecher la creation si un abonnement actif/essai/echec existe deja

### Tache 7 : API promo validate

`app/api/promo/validate/route.ts` :

- POST { code: string }
- Cherche le PromoCode dans Payload
- Verifie : actif, dans la periode de validite, currentUses < maxUses
- NE PAS incrementer currentUses ici (sera fait au checkout)
- Retourne : { valid: true, discountType, discountValue, appliesTo } ou { valid: false, message }
- Rate limiting basique (pas plus de 10 requetes/minute par IP — simple in-memory Map)
